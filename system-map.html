The prototype simulates the read path and UX shape only. Data is sample data, storage is the browser's localStorage, and there is no live Breeze connection. The "three connected accounts" are theater that mirrors the real poll-into-cache design.

4 · Architecture-critical dependencies

Only the questions that change the technical design appear here. The full open-questions list — with what each one unblocks — lives in the Ecosystem & Roles map.

  • Q11 — pivotal — Does each church's Tithely-unified plan still expose REST API access? The entire read-side rests on this; if not, that church needs a local-data fallback.
  • Q9 / Q10 — key topology — All three on Breeze or a subset, separate accounts or shared? Decides one API key vs. three, and the cache's isolation model.
  • No webhooks — confirmed; forces scheduled polling into the cache rather than event-driven sync (reflected in the diagram above).
